# 调试记录

## 2026-06-18 RGB LED 不亮

现象：板载 RGB LED 在流水彩虹固件下仍不亮。

已确认：

- 当前目标为 `esp32c5`，`CONFIG_BLINK_GPIO=27`。
- 固件使用 `led_strip` + RMT 后端驱动 WS2812 类 RGB LED。
- 原理图显示 RGB LED 为 `U4 WS2812E`，数据链路为 `GPIO27 -> Q3 2N7002 -> RGB_CTRL -> DIN`，LED 供电来自 `VCC_5V`。
- 2026-06-18 已烧录高亮诊断固件：启动后依次输出满白 5 秒、满红 1 秒、满绿 1 秒、满蓝 1 秒，然后进入满亮彩虹循环。
- 短时间串口监视确认固件已进入 `led diagnostic: full white/red/green/blue`，BLE 广播也正常启动。
- 重构前曾将 `led_set_rgb()` 改为返回 `esp_err_t`，启动诊断日志显示满白、红、绿、蓝四次发送结果均为 `ESP_OK (0)`；重构后对应底层接口为 `drv_rgb_led_set_rgb()`。这说明 ESP-IDF 的 `led_strip_set_pixel()` 和 `led_strip_refresh()` 调用本身没有报错。
- 曾加入单通道轮换 probe：启动诊断后每 5 秒依次把同一套 WS2812 数据输出到 `GPIO23`、`GPIO24`、`GPIO4`、`GPIO5`。串口日志确认四个候选 IO 均能启用并返回 `ESP_OK (0)`。
- 2026-06-18 已移除多 IO probe 轮换代码，当前固件只保留板载 RGB 控制链路：`CONFIG_BLINK_GPIO=27`。
- WSL2 USB/IP 下，`timeout` 包裹 `idf.py monitor` 可能留下 `idf_monitor.py` 子进程；监视结束后需用 `pgrep -af 'idf_monitor|esp_idf_monitor'` 检查，必要时手动 `kill`。

下一步：

- 如果实物仍完全不亮，优先测量 `VCC_5V`、`GPIO27`、`RGB_CTRL`。
- 预期电平：`GPIO27` 高电平约 3.3 V，`RGB_CTRL` 高电平约 5 V；低电平时两侧应接近 0 V。
- 逻辑分析仪到货后，优先抓 `GPIO27` 的 WS2812 数据波形；如需抓 `RGB_CTRL` 或 `DIN`，先确认逻辑分析仪输入是否支持 5 V 电平。
- 不应直接把板载 RGB 的软件 GPIO 随机改成其他 IO：原理图上板载 RGB 的数据输入只连到 `GPIO27 -> Q3 2N7002 -> RGB_CTRL`。换其他 IO 只能用于排针电平测量或外接 WS2812 验证，不能直接点亮板载 RGB。
- 若电源和电平链路正常，再尝试 RMT 输出反相或调整 WS2812 时序。
