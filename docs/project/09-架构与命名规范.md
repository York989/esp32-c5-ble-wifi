# 架构与命名规范

本文档定义本项目后续新增、移动、删除代码时必须遵守的分层、文件命名和符号命名规则。更短的强制规则同步写入根目录 `AGENTS.md`，用于新会话自动读取。

## 分层目标

项目采用 `app -> svc -> drv -> bsp` 四层结构，目标是高内聚、低耦合。

- `app`：应用编排层，只负责启动顺序、任务组合和模块生命周期。
- `svc`：业务服务层，负责 BLE 服务、LED 效果、心率服务等业务逻辑。
- `drv`：驱动适配层，负责 WS2812、GPIO、RMT 等硬件控制封装。
- `bsp`：板级支持层，负责开发板 IO 编号、硬件资源和板级常量。

依赖方向只能向下：`app -> svc -> drv -> bsp`。禁止 `drv` 调用 `svc`，禁止 `bsp` 依赖业务代码。

## 目录结构

当前阶段采用四个 ESP-IDF 组件目录，具体模块直接放 `.c` 和 `.h`，不再为每个模块拆 `include/` 与 `src/`。

```text
main/
  app_main.c

components/
  bsp/
    bsp_board.c
    bsp_board.h
    CMakeLists.txt

  drv/
    CMakeLists.txt
    rgb_led/
      drv_rgb_led.c
      drv_rgb_led.h

  svc/
    CMakeLists.txt
    led_effect/
      svc_led_effect.c
      svc_led_effect.h
    ble/
      svc_ble.c
      svc_ble_gap.c
      svc_ble_gatt.c
      svc_ble.h
    heart_rate/
      svc_heart_rate.c
      svc_heart_rate.h

  app/
    CMakeLists.txt
    ble_demo/
      app_ble_demo.c
      app_ble_demo.h
```

`main/app_main.c` 只调用应用层入口，例如 `app_ble_demo_start()`。

## CMake 规则

普通文件夹不需要 `CMakeLists.txt`。ESP-IDF 只有在某个目录要作为 component 被识别时，才需要组件级 `CMakeLists.txt`。因此本项目只在 `components/bsp/`、`components/drv/`、`components/svc/`、`components/app/` 放组件级 CMake 文件。

示例：

```cmake
file(GLOB_RECURSE srcs "*.c")

idf_component_register(
    SRCS "${srcs}"
    INCLUDE_DIRS "." "rgb_led"
    REQUIRES bsp
)
```

如果后续某一层变得过大，再考虑把其中的模块拆成独立 component；不要提前为每个小模块创建 CMake 文件。

## 文件与函数命名

文件名必须包含层级和模块名：

- `drv_rgb_led.c` / `drv_rgb_led.h`
- `svc_led_effect.c` / `svc_led_effect.h`
- `svc_ble_gap.c`
- `app_ble_demo.c`

公共函数使用 `<layer>_<module>_<action>()`：

```c
drv_rgb_led_init();
drv_rgb_led_set_rgb();
svc_led_effect_start();
svc_ble_start();
app_ble_demo_start();
```

私有 `static` 函数也保留模块前缀，例如 `svc_led_effect_rainbow_task()`，避免跳转到函数体后无法判断归属。

## 变量命名

局部变量使用明确语义，不加作用域前缀：

```c
uint8_t red;
esp_err_t err;
```

文件内私有静态变量使用 `s_<module>_<name>`：

```c
static led_strip_handle_t s_drv_rgb_led_strip;
static bool s_svc_led_effect_is_rainbow_enabled;
```

跨文件只读全局常量使用 `g_<module>_<name>`：

```c
extern const bsp_board_info_t g_bsp_board_info;
```

默认禁止跨文件可变全局变量。确实需要时必须使用 `g_` 前缀，并在头文件声明处写明所有权、线程保护方式和唯一写入入口。不要用 `volatile` 代替互斥、事件组或临界区；`volatile` 只用于 ISR 或硬件寄存器相关场景。

## 注释规则

头文件必须说明模块职责，公共函数必须说明用途、参数和返回值。注释重点解释“为什么这样做”和硬件、BLE、并发相关约束；不要写“设置变量”“调用函数”这类重复代码本身的注释。

## 迁移规则

重构时按模块逐步迁移，保持每一步可构建：

1. 先迁移 `bsp` 和 RGB LED 驱动。
2. 再把 LED 彩虹和启动诊断放入 `svc/led_effect/`。
3. 再迁移 BLE GAP/GATT 到 `svc/ble/`。
4. 再迁移心率 mock 到 `svc/heart_rate/`。
5. 最后让 `main/app_main.c` 只保留入口调用。

每次移动文件、改公共函数名、改 BLE 协议、改硬件 IO 或改构建方式，都要同步更新本文档或相关项目文档。
